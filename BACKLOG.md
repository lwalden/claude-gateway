### B-009: Authenticate or coarsen /health/cli — unauthenticated endpoint leaks credential expiry timing

Unauthenticated GET /health/cli returns subscription credential expiry timing (expiresAt epoch ms, hoursRemaining, status). Lets any networked caller learn when credentials expire — enables timing attacks toward the paid API path and fingerprints the gateway as subscription-backed. Medium severity (info disclosure). Options: (1) gate behind bearerAuth, (2) coarsen payload to status-only, (3) document-and-firewall (current). Keep GET /health open for liveness. Tracked: https://github.com/lwalden/claude-gateway/issues/30 — from self-review of PR #28.

### B-010: Validate proactive token refresh does not invalidate concurrent Claude CLI sessions (refresh-token rotation)

From triaging the abandoned chore/add-readme branch (commit 423a2ba, 2026-04-02). auth-refresh.js refreshes OAuth tokens on a 7.5h timer and writes the rotated refreshToken back to ~/.claude/.credentials.json. Concern: if Anthropic rotates the refresh token, concurrent Claude sessions (VS Code, terminal) that cached the old token may be invalidated. main deliberately KEPT proactive refresh and built auth-notify + the n8n expiry-alert workflow on top of it, so this was not adopted — but now that --bare is removed and the gateway actually uses the OAuth subscription, token freshness genuinely matters. Spike: confirm whether refresh-token rotation actually conflicts with concurrent sessions in current CLI v2.x; if so, decide between gateway-owned refresh vs letting each claude -p subprocess self-refresh. Note: refresh is a no-op in CONTAINER_MODE (no credentials file).
