# Product Brief & Roadmap

> The "why" and "what" behind this project. Claude references this on-demand for decision context.

---

## What & Why

**Problem:** Other local tools and scripts need to call Claude programmatically, but the Claude CLI is interactive-first and the Anthropic API costs per token. There's no simple HTTP interface that tries the subscription CLI first and falls back to the API.
**Vision:** A local HTTP gateway that any tool can `POST` to for Claude responses, using the subscription when possible and the API when not.
**Target Users:** The developer (single-user, local use) and their local tooling (scripts, automations, other services).

---

## MVP Features (Phase 1)

1. `/ask` endpoint -- Acceptance: POST with `{ prompt }` returns a Claude response via CLI or API fallback; response includes `source` field indicating which path was used
2. Bearer token auth -- Acceptance: requests without a valid `Authorization: Bearer` header receive 401; requests with a valid token succeed
3. CLI-first invocation -- Acceptance: when Claude CLI is available and responds within timeout, the CLI path is used (source: "cli"); when CLI fails or times out, the API path is used (source: "api")
4. Configurable behavior -- Acceptance: PORT, GATEWAY_API_KEY, ANTHROPIC_API_KEY, ANTHROPIC_MODEL, CLI_TIMEOUT_MS are all configurable via environment variables with sensible defaults
5. Security hardening -- Acceptance: no secrets in error responses, input validation on prompt field, safe handling of malformed requests

**Out of Scope:**
- Multi-turn conversation or session management (caller manages history)
- Streaming responses
- Multiple concurrent users or user identity
- Web UI or dashboard
- Database or persistence layer
- Cross-platform CLI invocation (Windows-only for now)
- Rate limiting or usage tracking

## Technical Stack

| Component | Choice | Why |
|-----------|--------|-----|
| Runtime | Node.js | Familiar, fast startup, good for I/O-bound gateway |
| Framework | Express | Minimal, well-known, sufficient for a thin proxy |
| Language | Vanilla JS | Small codebase (~160 lines), no build step needed |
| CLI integration | PowerShell EncodedCommand | Bypasses Windows quoting issues with `claude.cmd` |
| Auth | Shared-secret bearer token | Simplest viable auth for single-user local use |

## Phases

**Phase 1 (MVP):** Validate and harden -- test existing code, review configurables, security audit
**Phase 2:** Capabilities -- streaming responses, system prompt defaults, request logging, health check enhancements
**Phase 3:** Robustness -- cross-platform CLI support (Unix), retry logic, graceful degradation, optional rate limiting
